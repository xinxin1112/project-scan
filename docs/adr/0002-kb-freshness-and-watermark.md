# 0002 — 通过定时器 + 参考时间检查实现 KB 新鲜度，静默自动更新

## 背景

KB 描述的是主分支在特定 commit 时的快照。实际用户面临两个风险：

1. **本地过期**：开发者 3 天前拉取了 main，KB 仍反映那个旧 commit，AI 给出过时的回答。
2. **测试环境过期**（未来产品问答场景）：测试环境向产品人员提供过时的 KB，他们无法判断答案是否错误。

用户的原始需求："定时任务或引用时检查；如果超过半天没有更新，拉取 main 并检查；如果不是最新的，询问用户"。经过 grilling 后明确：

- "或"应该是**且**——两者都需要触发。
- 阈值调整为 **12 小时**。
- "询问用户"对产品问答消费者不适用（产品人员无法回答"我们是否应该更新 KB？"）。改为：静默自动更新并带过期水印，仅当用户运行 `/project-scan check` 时才显式提示。

## 决策

**两个触发器的 AND 条件，12 小时阈值，静默自动更新。**

- 定时任务：每 12 小时执行 `git fetch origin <main-branch>`，如果 main 有变动则增量更新。
- 参考时间检查：当 Agent（Claude Code 或未来的问答系统）即将读取 KB 时，skill 检查距上次新鲜度检查过了多久。如果 > 12 小时，同步执行 `git fetch` 并更新。
- 两条静默路径中都不进行交互式提示。fetch 或更新失败不阻塞——消费者继续使用现有 KB，但 frontmatter 标记为 `stale`，Agent 在回答中浮现警告。
- 功能分支保护：如果用户的本地工作树在一个偏离 main 的功能分支上，静默路径跳过自动更新并保留之前的 main 分支快照。消费者的心智模型是"KB = main 分支基线"，而非"KB = 我正在开发的内容"。
- `/project-scan check` 是显式路径——这里才适合提示用户。

**回答上的版本水印。** 任何引用 KB 的 Agent 回答都附加 `(KB version: <commit>, last update: <time>)`。如果过期超过 24 小时，Agent 额外在前面添加警告。这是测试环境的安全网：产品人员看到水印，可以质疑明显滞后于现实的回答。

## 后果

- 该 skill 必须在 `/project-scan` 设置期间注册一个 12 小时定时任务。各操作系统的实现已存在（`auto-update.js` / `setup-auto-update.js` 用于 launchd/Task Scheduler）——复用，将间隔调整为 12 小时。
- 每个 KB 文档需要一个 `last_scan_commits` 字段（已在 frontmatter 规格中锁定）。新鲜度检查将此与 `git rev-parse origin/<main-branch>` 进行比较。
- 消费者 Agent 需要知道如何渲染水印——这是添加到生成的 `<repo>/CLAUDE.md` 中的系统提示级指令（"引用 KB 内容时，包含版本水印"）。
- 未来的产品问答前端在构建时可免费继承此新鲜度模型。
