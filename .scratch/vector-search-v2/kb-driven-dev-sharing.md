## 1. 当前规范驱动开发，卡在哪

现有方案普遍绕不开一个问题：AI拿到的信息是**局部、非结构化的**，每一步都要猜。

### 三种常见方式，各卡在哪

```
方式一：SpecKit 逐步推断
┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐
│ 一句话需求 │ → │ AI澄清   │ → │ AI拆解   │ → │ AI对齐   │ → │ AI生成   │ → ❓
└──────────┘   └──────────┘   └──────────┘   └──────────┘   └──────────┘
     ↓              ↓              ↓              ↓              ↓
  信息最少      靠猜理解       理解再猜       两次猜测       累积5步误差
```

**问题**：每一步都可能偏差，5步下来误差放大。没有硬约束告诉AI"必须做成什么样"。

```
方式二：Superpowers 头脑风暴
┌──────────┐   ┌──────────┐   ┌──────────┐
│ 讨论需求  │ → │ AI确认理解│ → │ AI生成   │ → ❓
└──────────┘   └──────────┘   └──────────┘
     ↓              ↓              ↓
  内容非结构化    AI可能"说服"你   不知道怎么验证
```

**问题**：讨论是聊天，不是规范。你手里没有"什么是对的"的标准，AI说像就像。

```
方式三：把代码当知识库，加点注释
┌────┐ ┌────┐ ┌────┐ ┌────┐     ┌────────────────────┐
│Svc1│ │Svc2│ │Mq1 │ │Ctrl│ →  │ AI拼10个文件出全局图 │ → ❓
└────┘ └────┘ └────┘ └────┘     └────────────────────┘
                                        ↓
                              信息碎片化，一定丢东西
```

**问题**：分散在N个文件里，人和AI都拼不出全局全貌。不是读不懂代码，是信息不在一起。

### 最终导向同一个死结

> **人Review AI生成的代码：**
>
> "看起来挺像那么回事" — 但漏了一个边界条件？少处理了一种回调状态？有个表忘了同步更新？
>
> **人没法判断。AI没法自证。**
>
> 没有统一的判断标准 → **心里没底 → 不敢用。**

### 核心矛盾

| 缺什么 | 为什么致命 |
|--------|-----------|
| 共享的全局认知 | 人和AI各自理解，不在一个频道 |
| 可验证的硬标准 | 不知道"对"长什么样，Review全靠猜 |
| 完成的定义 | 不知道什么时候算做完、做完对不对 |

## 2. 方案：把"全局知识"显式化、结构化

### 核心理念

```
不是"在代码里补文档"，
而是"文档先行，代码必须贴合文档"。
文档是验收标准，不是辅助注释。
```

### 四层结构

```
docs/kb/
├── domain/              ← 数据与规则（20+个文件）
│   ├── entities/          DDL + 字段 + 索引 + 约束 + 关联
│   ├── enums/             枚举定义
│   ├── state-machines/   状态转移图 + 守卫条件 + 副作用
│   └── rules/            决策表：什么时候判定为PAY_OVERSEAS
│
├── contracts/           ← 接口契约（5个文件）
│   ├── internal/         内部服务间：请求/响应/异常/幂等
│   └── external/         外部回调：Flow引擎、HLY
│
├── flows/               ← 流程编排（6个文件）
│   └── overseas-payment/ 5步流程：每步输入/输出/子步骤/异常
│
└── code/                ← 实现索引（1个文件）
    └── method-index.md   方法→行号→类→职责映射
```

### 每一层只回答一个问题

| 层 | 回答 |
|----|------|
| domain | 有什么数据？怎么变的？有什么规则？ |
| contracts | 谁调谁？传什么？回什么？ |
| flows | 分几步？每步做什么？异常怎么处理？ |
| code | 实现在哪个类的哪个方法？ |

### 为什么这能解决"验收标准"问题

```
生成代码 → 对照知识库校验：
  ✅ 实体字段都对吗？
  ✅ 索引都建了吗？
  ✅ 状态转移的副作用都执行了吗？
  ✅ 流程步骤的顺序对吗？
  ✅ 异常处理覆盖了吗？

有标准 → 能判断对错 → 人和AI都不恐慌
```

## 3. 示例：海外付款——顺一条链路看全貌

### 不是"把代码翻译成人话"，是"跨层拉一条线"

以 **"HLY回调付款成功"** 这一条真实指令为例，知识库中 4 层怎么协作：

```
HLY系统推送回调：
{ businessCode: "PY2505140001", expenseStatus: 105 }

        ┌──────────────────────────────────────────────────┐
  ①     │ contracts/external/hly-status-callback.md        │
 contract│ expenseStatus=105 → PAYMENT_SUCESS               │
        │ businessCode → 匹配 initiate.pay_number           │
        │ 决定：走 success() 处理，不是 reject/log/skip     │
        └──────────────────────┬───────────────────────────┘
                               │ 输入确认，进入流程
                               ▼
        ┌──────────────────────────────────────────────────┐
  ②     │ flows/step-4-callback.md                          │
  flow  │                                                    │
        │ 执行序列:               数据流向:                  │
        │ ① 防重检查              initiate_status           │
        │ ② 动转确 commitment     APPROVE → SUCCESS         │
        │ ③ 更新付款单状态         payment_status            │
        │ ④ 机器人审批资金节点     — → HAS_PAY              │
        └──────────────────────┬───────────────────────────┘
                               │ 步骤+字段变更全标了
                               ▼
        ┌──────────────────────────────────────────────────┐
  ③     │ domain/state-machines/initiate-lifecycle.md       │
 domain │                                                    │
        │ APPROVE ──→ SUCCESS                               │
        │ 守卫: expenseStatus=PAYMENT_SUCESS(105)           │
        │ 副作用: commitment(预算占用→实际支出)              │
        │ 副作用: 回写 HLY payNumber                        │
        └──────────────────────┬───────────────────────────┘
                               │ 规则也定了，可以写了
                               ▼
        ┌──────────────────────────────────────────────────┐
  ④     │ code/overseas-payment/method-index.md             │
  code  │                                                    │
        │ PayOverSeasCallbackServiceImpl.callback() — 入口  │
        │ PayOverSeasCallbackServiceImpl.success()  — 实现  │
        │ → app/.../callback/PayOverSeasCallbackImpl:125+   │
        └──────────────────────────────────────────────────┘
```

### 这和"读代码"有什么本质区别

| | 读代码 | 读知识库 |
|----|--------|---------|
| **状态码处理** | 看到 if/else → 不知道覆盖全没 | 契约列了 8 种状态码，哪些处理、哪些跳过、哪些仅记录，一眼全 |
| **字段变更** | 看到 setXxx() → 不知道为什么修改 | 数据流向标注 "变更前→变更后→条件"，跃迁规则透明 |
| **副作用** | 散落在 3 个 Service 里 → 拼不齐 | 状态机显式声明：commitment、机器人审批、bill释放 |
| **全局拼图** | 跨 5 个类 + 2 个外部接口 | 一条线从 contract → flow → domain → code 拉到底 |

### 全局可追溯的结构本质

```
任何业务链路，4 层结构是统一的：

contracts（谁调我、传什么）
    ↓
flows（分几步执行、数据怎么映射）
    ↓
domain（什么规则、数据怎么变）
    ↓
code（实现在哪个类的哪个方法）
```

这不是"给代码配说明"，而是**代码必须贴合这 4 层定义**。业务、产品看 contracts+flows，开发看 flows+domain+code，AI 看 4 层全量。同一份认知，不靠猜。

## 4. 对比：为什么这种方式可能更精准

### 4.1 与 SpecKit / Superpowers 的本质区别

| 维度 | SpecKit/传统 | 本方案 |
|------|-------------|--------|
| **起点** | 一句话需求（信息量低） | 完整领域知识库（信息量高） |
| **过程** | 逐步推断，误差累积 | 从硬约束出发，不靠推断 |
| **验收** | 没有统一标准 | 代码必须贴合文档，可逐项核验 |
| **修复** | 歪了重来，代价大 | 多Agent交叉校验 + 定向修复 |
| **全局观** | AI拼凑，人类也拼不出 | 人类和AI共享同一份显式全局知识 |

### 4.2 精准度的来源：不是AI更强，是约束更严

```
传统：AI根据"创建付款单"这个需求 → 推断需要什么字段 → 生成代码
      （推断过程中可能漏字段、漏约束、漏关联）

本方案：AI已有 initiate.md 的完整定义 →
      字段60+个、4个索引、5个关联实体、状态转移副作用全有了 →
      AI的工作是"翻译成代码"，不是"推断业务逻辑"
```

### 4.3 为什么人更容易理解

- **分层清晰**：先看流程理解业务，再看实体理解数据，最后看代码定位实现
- **关联显式**：实体间关系用表格标注，不是靠代码import反推
- **决策透明**：规则用决策表，不是用if-else代码块
- **状态可视化**：Mermaid图 + 守卫条件 + 副作用表，不用读几十个状态变更代码

## 5. 验证：歪了怎么办

### 不是"一次性生成"，是"多Agent交叉校验 → 修复 → 收敛"

```
┌──────────────┐
│  知识库规范   │ ← 验收标准（唯一真理源）
└──────┬───────┘
       ▼
┌──────────────┐     ┌──────────────┐
│  Agent A     │     │  Agent B     │
│  生成代码     │     │  校验一致性   │
└──────┬───────┘     └──────┬───────┘
       │                    │
       └────────┬───────────┘
                ▼
        ┌──────────────┐
        │  发现偏差？    │──Yes──▶ 修复 → 再来一轮
        └──────┬───────┘
               │No
               ▼
        ┌──────────────┐
        │  通过验收     │
        └──────────────┘
```

**为什么能收敛：**
- 验收标准是确定的（文档中的字段、约束、状态转移）
- 多Agent避免单点AI能力偏差
- 偏差是"代码和文档不一致"，不是"人类觉得不对"
- 最终状态：代码100%贴合文档 → 就算系统逻辑有问题，也是文档没描述清楚

## 6. 未来：两个方向

### 6.1 产品-开发知识联动

```
现在：开发知识库（技术视角：DDL、接口、状态机、流程步骤）

未来：+ 产品知识库（业务视角：功能需求、验收条件、业务场景）
       ↓
       两个视角的知识关联 →
       产品提需求时，AI基于开发知识库判断"信息是否足够开发"
       →
       AI给出设计方案 + 修改建议 →
       开发确认 → 生成代码 → 产品验收
```

**价值：降低产品与开发的沟通gap**
- 产品不需要理解DDL，但AI知道"这个字段关联什么约束"
- 开发不需要反复追问需求细节，AI已经基于知识库做了预判
- 同一个知识平台 → 同一套认知 → 协作效率

### 6.2 知识库自动进化

```
需求变更 → 开发完成 → 生成变更摘要 →
                              ↓
                     更新知识库（新增/修改实体、流程、契约）
                              ↓
                     归档变更历史 →
                     下次需求来时，知识库已是最新
```

闭环：**开发在写代码，知识库在长**

### 6.3 定制工作界面

```
┌──────────────────────────────────────────────┐
│ 左侧：业务/产品知识库 │  中间：AI协作区       │
│ ─────────────────── │  ──────────────────  │
│  需求文档            │  ① 业务/产品提出需求   │
│  业务场景            │  ② AI判断信息完整度    │
│  验收条件            │     （参考开发知识库）  │
│                      │  ③ AI输出方案+建议    │
│ 右侧：开发知识库      │  ④ 开发确认+调整      │
│ ─────────────────── │  ⑤ 生成代码          │
│  实体 DDL            │  ⑥ 校验通过 → 归档    │
│  状态机              │  ⑦ 更新知识库        │
│  接口契约            │                      │
│  代码索引            │                      │
└──────────────────────────────────────────────┘
```

**这套界面的核心：**
不是单个工具的提效，是**把业务、产品、开发、AI拉到同一个认知平面上协作**。
各方操作自己的知识库视角，AI居中判断"业务/产品给的信息是否足够支撑开发"。

## 7. 目前不足 & 下一步

### 当前局限

| 不足 | 说明 |
|------|------|
| **规范粒度还在探索** | 太多 → 构建成本高；太少 → 信息不够。需要找到"刚好能指导代码生成"的平衡点 |
| **外部系统表达** | 知识库目前以内部领域为主，外部系统（Flow/HLY）的交互细节还需要更规范的表达 |
| **知识库维护成本** | 谁来更新？如何保证质量？目前靠人类校验，缺乏自动化手段 |
| **跨领域关联** | 产品知识和开发知识的关联机制还在设计阶段，尚未落地 |
| **未完全成型** | 目前是一个概念验证阶段，尚未大规模验证可行性 |

### 下一步

1. **规范打磨**：在实际开发中校准知识库粒度，形成"最佳实践模板"
2. **自动化校验**：建立知识库质量检查工具（如实体文档是否覆盖所有DDL字段、流程步骤是否覆盖所有代码路径）
3. **Agent联调**：用不同AI模型做交叉验证，测试收敛效果和修复成功率
4. **产品知识库试点**：选取一个业务场景，建立产品视角的知识库并与开发知识库关联
5. **持续进化闭环**：自动化变更检测 → 知识库更新 → 版本归档

---

## 总结

```
这个方案的底层逻辑：

不是让AI更聪明，
而是给AI和人类提供同一份结构化、可验证的全局知识，
让生成代码"有标准可依"、让验收"有据可查"、让修复"有方向可循"。

它不是单点工具提效，
而是将业务、产品、开发拉到同一个认知平台上的协作升级。
```
